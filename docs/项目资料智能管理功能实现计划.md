# 项目资料智能管理功能实现计划

**1. 目标**

在现有 `DRFForVue` 后端和 `calendar_with_react` 前端基础上，实现项目资料智能管理功能，包括：
*   **文件分类管理**：支持多类型文件（PDF、Word、图片等）的项目归档和分类。
*   **智能分析与查漏补缺**：利用大语言模型（LLM）能力，识别文档中的不规范、时间冲突、内容缺失、内容与规定不符等问题。
*   **页面提醒**：在前端页面直观展示不合规提醒。
*   **定时提醒**：对有时间要求的规范文件进行定时检查和页面提醒。
*   **项目管理**：将文件和不合规项与具体项目关联，实现项目档案专家职能。

**3. 详细实现步骤**

**3.1 后端 (DRFForVue)**

*   **3.1.1 文件预处理（OCR 和文本提取）**
    *   **目标**：从 PDF、Word、图片等文件中提取可分析的文本内容。
    *   **操作**：
        *   **优先使用 Mineru 服务进行 OCR**：
            *   集成 Mineru API：通过 HTTP 请求将图片文件发送给 Mineru 服务进行 OCR 识别，获取文本内容。
            *   对于 PDF 和 Word 文档：可以先将其转换为图片格式（如果 Mineru 直接支持 PDF/Word 解析则直接调用），再调用 Mineru 进行 OCR。或者，对于文本可复制的 PDF/Word，继续使用 `PyPDF2`、`python-docx` 等库直接提取文本。
        *   **引入必要的库**：
            *   对于 PDF：考虑使用 `PyPDF2` 或 `pdfminer.six`。
            *   对于 Word 文档：使用 `python-docx`。
        *   **修改 `documents/views.py` 中的文件上传逻辑**：在文件上传后，根据文件类型调用相应的文本提取模块（优先 Mineru OCR），将提取出的文本作为后续智能分析的输入。

*   **3.1.2 项目管理功能**
    *   **目标**：实现项目档案的统一管理，将文件和不合规项归属到特定项目。
    *   **操作**：
        *   **新增 `projects` 应用**：
            *   创建 `projects/models.py`：定义 `Project` 模型，包含字段如 `name` (项目名称), `description` (项目描述), `start_date` (开始日期), `end_date` (结束日期), `manager` (关联 `User` 模型，项目负责人), `status` (项目状态，如进行中/已完成) 等。
            *   创建 `projects/serializers.py`：为 `Project` 定义序列化器。
            *   创建 `projects/views.py`：为 `Project` 定义 `ViewSet`，提供项目的增删改查接口。
            *   在 `DRFForVue/DRFForVue/urls.py` 中添加 `path('projects/', include('projects.urls'))`。
            *   在 `DRFForVue/DRFForVue/settings.py` 的 `INSTALLED_APPS` 中添加 `'projects'`。
        *   **关联现有模型**：
            *   修改 `documents/models.py`：在 `Book` 和 `DocumentTemplate` 模型中添加一个 `ForeignKey('projects.Project', on_delete=models.CASCADE, related_name='documents')` 字段。
            *   创建新的 `compliance` 应用（如果尚未创建）：
                *   创建 `compliance/models.py`：定义 `ComplianceIssue` 模型，包含字段如 `project` (ForeignKey 到 `projects.Project`), `document` (ForeignKey 到 `documents.Book` 或 `DocumentTemplate`), `issue_type` (不规范/时间冲突/内容缺失/内容与规定不符), `description`, `location` (页码/段落/图片区域), `status` (待处理/已解决), `severity`, `due_date` (针对有时间要求的规范)。
                *   创建 `compliance/serializers.py`：为 `ComplianceIssue` 定义序列化器。
                *   创建 `compliance/views.py`：为 `ComplianceIssue` 定义 `ViewSet` 或 APIView，提供不合规项的增删改查接口，并确保查询可以按项目过滤。
            *   在 `DRFForVue/DRFForVue/settings.py` 的 `INSTALLED_APPS` 中添加 `'compliance'`。
            *   在 `DRFForVue/DRFForVue/urls.py` 中添加 `path('compliance/', include('compliance.urls'))`。
        *   **调整 API 逻辑**：在文件上传、不合规项生成等操作中，确保能够接收并关联到相应的 `project_id`。

*   **3.1.3 智能分析与查漏补缺（利用现有 LLM 服务）**
    *   **目标**：通过调用现有大语言模型（如 Dify AI 或 Ollama 部署的模型）进行智能分析，识别不规范、时间冲突、内容缺失、内容与规定不符等问题。
    *   **操作**：
        *   **修改 `documents/views.py`**：
            *   在 `DocumentTemplateViewSet` 的 `analyze_file` 方法中，替换模拟逻辑。
            *   **核心逻辑**：将预处理后的文档文本（或特定段落、章节）作为输入，构建合适的提示词（Prompt），发送给大语言模型进行分析。
            *   **提示词设计**：提示词应明确指示 LLM 扮演“规范审查员”的角色，要求其：
                *   识别文本中的“不规范”之处（例如，格式错误、用词不当）。
                *   识别“时间冲突”（例如，项目开始日期晚于结束日期，或与重要截止日期冲突）。
                *   识别“内容缺失”（例如，合同中缺少关键条款，或报告中缺少必要数据）。
                *   识别“内容与规定不符”（例如，与已知的某个规范条款相悖）。
                *   LLM 的输出应结构化（例如，JSON 格式），包含问题类型、描述、建议修改、相关文本片段、置信度等信息，以便后端解析并存入 `ComplianceIssue` 模型。
            *   **集成 LLM**：
                *   **Dify AI**：继续利用 `office_assistant` 应用中已有的 `requests` 库进行 API 调用。可以为不同的“问题类型”在 Dify 中创建不同的应用或 Agent，或者设计一个通用的 Agent 来处理多种问题。
                *   **Ollama**：在 `DRFForVue/office_assistant/views.py` 或创建一个新的服务层来管理 Ollama 的调用，通过 HTTP 请求与其 API 交互。
        *   **规范知识库（RAG 辅助）**：
            *   将规范文档切块并向量化存储（可以考虑使用 `pgvector` 这样的向量数据库扩展）。
            *   在分析时，根据待分析文档的内容，从规范知识库中检索最相关的规范片段。
            *   将检索到的规范片段与待分析文档一起作为 LLM 的输入，引导 LLM 更准确地判断合规性。

*   **3.1.4 定时提醒功能**
    *   **目标**：对于有时间要求的规范文件，实现定时检查和页面提醒。
    *   **操作**：
        *   安装并配置 `celery` 和 `django-celery-beat`。
        *   在 `settings.py` 中配置 Celery Broker (如 Redis) 和 Celery Beat。
        *   在 `compliance` 应用中创建 `tasks.py`：
            *   编写 Celery 任务，定期扫描 `ComplianceIssue` 模型中 `due_date` 临近或已过期的项。
            *   任务触发时，更新 `ComplianceIssue` 状态或创建一个新的 `Notification` 记录（如果存在单独的通知系统），确保前端可以通过 API 拉取这些提醒信息。

**3.2 前端 (calendar_with_react)**

*   **3.2.1 项目管理与文件展示**
    *   **目标**：提供项目管理界面，并在文件和不合规项展示中体现项目归属。
    *   **操作**：
        *   **新增 `src/pages/ProjectsPage.jsx`**：
            *   用于展示项目列表、创建新项目、编辑项目信息。
            *   每个项目可以点击进入其详情页，查看该项目下的文件和不合规项。
            *   新增 `src/api/projects.js` 用于与后端 `projects` API 交互。
        *   **修改文件列表页面** (`src/components/DocumentsPage.jsx`, `src/components/LibraryPage.jsx`)：
            *   增加按项目筛选的功能（例如，一个项目选择器）。
            *   在文件列表中显示所属项目信息。
        *   **修改不合规项列表页面** (`src/components/CompliancePage.jsx` 或新增页面)：
            *   增加按项目筛选的功能。
            *   在提醒列表中显示所属项目信息。
        *   **文件详情页和提醒详情页**：确保在这些页面中能够清晰地展示文件或提醒所属的项目信息。

*   **3.2.2 页面提醒展示**
    *   **目标**：在页面上直观、及时地展示各类提醒。
    *   **操作**：
        *   **通知中心**：在导航栏或侧边栏 (`src/components/Sidebar.jsx`) 添加一个通知图标，显示未读提醒数量。点击后跳转到提醒列表页面。
        *   **提醒列表页面**：开发一个新的页面（例如 `src/pages/NotificationsPage.jsx`），用于展示所有不合规和定时提醒。
            *   列表项包含提醒类型、描述、相关文件链接、严重程度、状态等。
            *   提供筛选、排序、标记已读、归档等功能。
        *   **文件详情页集成**：在 `src/components/BookReaderPage.jsx` 或 `src/components/ChapterView.jsx` 中，如果当前文件或章节存在不合规项，应在页面上以醒目方式（如高亮、警告图标）提示，并提供链接或弹窗查看详细问题。
        *   **利用 `src/utils/notifications.js`**：扩展其功能，来处理和展示前端通知。
