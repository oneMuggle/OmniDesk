## 2. 改进项明细

### I-1 修复 worker 启动 race condition

**根因分析:**
- `deployment/docker/omni_desk_backend/Dockerfile:103`:`ENTRYPOINT ["./entrypoint.sh"]` —— backend 和 worker 共用同一镜像
- `omni_desk_backend/entrypoint.sh:39-40`:无条件执行 `python manage.py migrate --noinput`
- `compose/docker-compose.offline.yml:140`:worker 的 `command: celery -A omni_desk_backend worker -l info` 通过 entrypoint.sh 的 `exec "$@"`(行 51)启动
- 结果:worker 和 backend 启动时同时跑 migrate,Postgres 报 `UniqueViolation: django_migrations_id_seq`

**修复方案:** 最小侵入式 + 向后兼容

1. 在 `entrypoint.sh` 顶部增加 SKIP_MIGRATE 检测:
   ```bash
   # 跳过 migrate / collectstatic(给 celery worker 等非 backend 服务用)
   if [ "${SKIP_MIGRATE:-false}" = "true" ]; then
       echo "SKIP_MIGRATE=true: 跳过 migrate / collectstatic"
       export HOME=/home/app
       exec setpriv --reuid=app --regid=app --init-groups "$@"
   fi
   ```
2. `compose/docker-compose.offline.yml` worker service 增加 `SKIP_MIGRATE: "true"`
3. 默认行为不变(无 SKIP_MIGRATE 时仍跑 migrate)

**验证标准:**
- worker 启动日志**无** `UniqueViolation` / `MigrationSchemaMissing`
- celery `status` 仍 online
- debug_task 仍能在 1s 内执行

## 4. 风险评估

| 风险 | 缓解 |
|---|---|
| SKIP_MIGRATE 被错误启用导致 backend 不跑 migrate | 默认 false + 仅 worker 配置 true |
| 离线包升级时 worker 仍用旧镜像 | 步骤 5 强制重建 |
| 修改 entrypoint.sh 影响其他场景(如 dev compose worker) | dev compose 通常不用 worker |
