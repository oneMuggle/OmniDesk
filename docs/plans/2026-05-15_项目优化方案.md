## Phase 1: 依赖版本对齐 + 安全加固 (4-6 小时)

### 1.1 依赖版本对齐 (2 小时)

#### 步骤 1: 诊断版本差异

```bash
cd omni_desk_backend
diff requirements.txt requirements-prod.txt
diff requirements.txt requirements-dev.txt
```

#### 步骤 2: 更新 source 文件

**涉及文件**: `requirements.in`

核心依赖升级计划:

| 包 | 当前锁定 (dev/prod) | 目标版本 | 升级原因 |
|---|---|---|---------|
| Django | 4.2.11 / 4.2.30 | >=4.2.30 | CVE 安全修复 |
| Gunicorn | 20.1.0 | >=23.0.0 | 性能改进 + 安全修复 |
| psycopg2-binary | 2.9.9 | 2.9.10 | Bug 修复 |
| redis | 4.6.0 | 5.2.x | 连接池改进 |
| celery | 5.5.3 | 5.5.x | 稳定性 |

#### 步骤 3: 重新编译锁文件

```bash
pip-compile requirements.in -o requirements.txt
pip-compile requirements.in requirements-dev.in -o requirements-dev.txt
pip-compile requirements.in -o requirements-prod.txt
```

- **风险**: 低。Django 4.2.x 内小版本升级向后兼容。
- **影响范围**: 全部模块 (基础依赖变更)
- **验收标准**: 三个 lockfile 中 Django/Gunicorn 版本一致，`pip install -r requirements.txt` 成功

### 1.2 安全加固 (2-4 小时)

#### 步骤 1: 审计 API 端点权限

遍历所有 `views.py`，检查每个 ViewSet 的 `permission_classes`:

| 文件 | 检查项 | 修复措施 |
|------|--------|---------|
| `users/views.py` | 注册/登录端点是否允许匿名 | 正常，但需加 Rate Limit |
| `documents/views.py` | 文档访问是否需要所有权验证 | 添加 Object-level permission |
| `smart_assistant/views.py` | LLM 调用是否有配额限制 | 添加用户级别限流 |
| `notifications/views.py` | 用户是否只能看自己通知 | 重写 `get_queryset` |
| `permissions/views.py` | 权限管理是否需管理员 | 添加 `IsAdminUser` |
| `dashboard/views.py` | 仪表盘数据是否越权 | 按用户/角色过滤 |

#### 步骤 2: 检查 CORS 配置

**涉及文件**: `omni_desk_backend/settings/local.py`
- 当前: `CORS_ALLOW_ALL_ORIGINS = True` (过度宽松)
- 修复: 改为 `CORS_ALLOWED_ORIGINS` 白名单

```python
# local.py 修改
CORS_ALLOWED_ORIGINS = [
    "http://localhost:3000",
    "http://127.0.0.1:3000",
]
```

**涉及文件**: `omni_desk_backend/settings/production.py`
- 确认 `CORS_ALLOWED_ORIGINS` 仅包含生产域名

#### 步骤 3: 添加 Rate Limiting

安装: `pip install django-ratelimit`

**涉及文件**:
- `requirements.in` — 新增 `django-ratelimit>=4.1.0`
- `omni_desk_backend/settings/base.py` — 添加 `"ratelimit"` 到 `INSTALLED_APPS`

关键端点装饰器:

| 端点 | 限流规则 | 文件 |
|------|---------|------|
| 登录 | `5/m` | `users/views.py` |
| 注册 | `3/h` | `users/views.py` |
| LLM 调用 | `20/h` | `smart_assistant/views.py` |

#### 步骤 4: 敏感信息审计

```bash
grep -rn "secret\|password\|api_key\|token" --include="*.py" | grep -v "settings\|test\|\.env"
```

确认以下文件在 `.gitignore`:
- `.env.local`
- `.env.production`
- `*.key`
- `db.sqlite3`

- **风险**: 中。CORS 收紧可能导致前端开发需额外配置。
- **影响范围**: 所有 API 端点
- **验收标准**:
  - 所有 ViewSet 有明确 `permission_classes`
  - CORS 无 `ALLOW_ALL_ORIGINS = True`
  - 登录/注册端点有 Rate Limit
  - 无硬编码密钥通过审计

## Phase 2: 大文件拆分 + 服务层提取 (12-16 小时)

### 2.1 documents/views.py 拆分 (4-5 小时)

**当前**: 362 行，混合 Template/Document/Book/Chapter/EBook 五个领域

**目标结构**:

```
documents/
├── views/
│   ├── __init__.py
│   ├── templates.py      # ~80 行 - DocumentTemplateViewSet
│   ├── documents.py      # ~80 行 - GeneratedDocumentViewSet
│   ├── books.py          # ~100 行 - Book/Chapter/ViewSet
│   └── ebooks.py         # ~60 行 - EBookViewSet
├── views.py              # 仅保留 re-export (向后兼容)
├── serializers.py        # 拆分为对应领域
├── services.py           # 新增 - 业务逻辑
├── tasks.py              # Celery 异步任务
├── models.py
└── urls.py
```

**实施步骤**:
1. 创建 `documents/views/` 目录
2. 按模型类型拆分 ViewSet 到独立文件
3. 提取文档生成逻辑到 `documents/services.py`:
   - `generate_document(template, data)`
   - `compose_book_chapters(book)`
   - `convert_to_ebook(document)`
4. 在 `views.py` 保留 re-export 保持 URL 引用不变
5. 更新 `documents/urls.py` 的 import 路径

- **风险**: 低。纯重构，URL 路由不变。
- **验收标准**: `python manage.py check` 通过，所有现有测试通过

## Phase 4: CI/CD + 类型检查 (16-20 小时)

### 4.1 GitHub Actions CI 流水线 (8-10 小时)

**新建文件**: `.github/workflows/ci.yml`

```yaml
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: pip install ruff
      - run: ruff check .
      - run: ruff format --check .

  test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:14
        env:
          POSTGRES_PASSWORD: test
      redis:
        image: redis:7
    steps:
      - uses: actions/checkout@v4
      - run: pip install -r requirements-dev.txt
      - run: pytest --cov --cov-fail-under=80 --cov-report=xml
      - uses: codecov/codecov-action@v4
        with:
          file: ./coverage.xml

  security:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install bandit safety
      - run: bandit -r . -ll
      - run: safety check -r requirements.txt
```

**流水线规则**:
- PR 必须通过 lint + test + security 检查
- coverage < 80% 时失败
- main 分支推送触发部署 (可选)

### 4.2 类型检查引入 (6-8 小时)

#### 步骤 1: 安装 mypy

```bash
pip install mypy types-requests types-redis types-python-dateutil
```

#### 步骤 2: 配置文件

**新建文件**: `mypy.ini`

```ini
[mypy]
python_version = 3.11
warn_return_any = True
warn_unused_configs = True
disallow_untyped_defs = False  # 渐进式引入

[mypy-django.*]
ignore_missing_imports = True

[mypy-rest_framework.*]
ignore_missing_imports = True

[mypy-celery.*]
ignore_missing_imports = True
```

#### 步骤 3: 渐进式添加类型注解

| 优先级 | 层级 | 原因 |
|--------|------|------|
| 1 | `services/` | 业务逻辑最重要，且是新文件 |
| 2 | `models.py` | 数据模型定义清晰 |
| 3 | `serializers.py` | 数据验证边界 |
| 4 | `views.py` | HTTP 层 |
| 5 | `tasks.py` | 异步任务 |

#### 步骤 4: CI 集成

在 `ci.yml` 新增 `typecheck` job:

```yaml
  typecheck:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install mypy types-requests
      - run: mypy . --ignore-missing-imports
```

- **验收标准**: mypy 无 error 级别报错
