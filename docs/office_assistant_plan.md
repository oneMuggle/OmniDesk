# Office文档助手模块开发计划

## 1. 目标

在现有项目中构建一个Office文档辅助模块，利用大语言模型（LLM）实现文档的智能纠错、翻译和润色功能。

## 2. 核心方案对比

| 特性 | **方案一：上传并处理文档 (Word/PDF)** | **方案二：集成Dify应用** |
| :--- | :--- | :--- |
| **核心优势** | 直接处理用户源文件，用户体验最直接。 | 开发速度快，AI能力（如模型、Prompt）的维护和迭代与主项目解耦，非常灵活。 |
| **核心挑战** | **格式保持极其困难**。解析Word/PDF，修改文本后，要完美无损地写回原格式，尤其是PDF，技术上非常复杂。 | 需要用户将文档内容**复制/粘贴**到我们的编辑器中。 |
| **技术复杂度** | **非常高**。 | **中等**。 |
| **开发周期** | **长**。 | **短**。 |
| **灵活性** | **低**。 | **高**。 |

## 3. 结论与建议

**结论**：直接处理文档的方案（尤其是保持格式）风险高、周期长。集成Dify应用的方案更加敏捷、务实，能让我们快速交付核心价值。

**建议**：我们采用 **“分阶段、逐步迭代”** 的策略，以 **“Dify集成 + 富文本编辑器”** 为起点，快速上线核心功能，然后根据反馈逐步增强文件处理能力。

## 4. 分阶段实施计划

### 第一阶段 (MVP): 实现核心AI文本处理功能

此阶段的目标是让用户能够在一个类Word的在线编辑器中，对文本进行智能处理。

**架构图:**
```mermaid
graph TD
    subgraph "前端 (calendar_with_react)"
        A[用户] --> B{富文本编辑器 (Tiptap/TinyMCE)};
        B -- 粘贴或输入文本 --> B;
        B -- 点击“润色”等按钮 --> C[触发API调用];
        C -- 1. 发送编辑器内容 --> D[Office助手API];
        D -- 4. 返回处理结果 --> C;
        C -- 5. 更新编辑器内容 --> B;
    end

    subgraph "后端 (DRFForVue)"
        D -- 2. 调用Dify API --> E[Dify平台];
    end
    
    subgraph "外部服务 (Dify)"
        E -- 预设的Prompt和LLM --> E;
        E -- 3. 返回处理后的文本 --> D;
    end

    F[在Dify创建应用] -- 管理和维护 --> E;
    A -- 管理 --> F[例如: 纠错App, 翻译App];
```

**详细步骤:**

1.  **前端 (React)**:
    *   集成一个功能强大的富文本编辑器（如 Tiptap 或 TinyMCE），提供基础的文本格式化能力。
    *   在编辑器工具栏旁增加“智能纠错”、“翻译”、“润色”等功能按钮。
    *   创建API服务，用于将编辑器中的内容发送到后端。
    *   实现点击按钮后调用API，并将返回结果更新回编辑器的逻辑。

2.  **后端 (Django)**:
    *   创建一个新的API端点，例如 `/api/office_assistant/process/`。
    *   此端点接收前端发来的文本内容和操作类型（如 `proofread`）。
    *   根据操作类型，调用在Dify平台上预先创建好的相应应用的API。
    *   将从Dify获取的结果直接返回给前端。

3.  **Dify平台**:
    *   创建至少三个应用：
        *   **智能纠错应用**：精心设计Prompt，专注于修正语法、拼写和标点错误。
        *   **文本翻译应用**：设计支持多种语言互译的Prompt。
        *   **商务润色应用**：设计Prompt，将普通文本改写得更专业、更具说服力。

### 第二阶段: 优化体验 - 支持文档文本导入

此阶段解决手动复制粘贴的痛点。

*   **前端**: 增加一个文件上传控件，允许用户上传 `.docx` 和 `.pdf` 文件。
*   **后端**:
    *   在API中增加文件处理逻辑。
    *   使用 `python-docx` 库解析 `.docx` 文件，提取全部文本。
    *   使用 `PyMuPDF` 库解析 `.pdf` 文件，提取全部文本。
    *   将提取出的纯文本返回给前端，自动填充到富文本编辑器中。
    *   **注意**: 此阶段我们只提取文本，不关心格式。

### 第三阶段 (可选/未来): 实现文档导出功能

在核心功能得到市场验证后，再投入资源解决格式保持问题。

*   **后端**:
    *   研究并实现将编辑器处理后的内容（HTML格式）写回到 `.docx` 格式的方案。
    *   提供一个“下载处理后的文档”按钮。

通过以上分阶段计划，我们可以在最短时间内上线一个可用、有价值的产品，同时将最复杂的技术难题留到后面，有效控制项目风险。
